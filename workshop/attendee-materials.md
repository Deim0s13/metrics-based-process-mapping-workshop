# Attendee Materials: Metrics Based Process Mapping Workshop

These materials support the **Metrics Based Process Mapping for Automation Discovery** workshop.

Use them during the session to follow the method, capture notes, and assess candidate automation opportunities.

> **Map the work. Measure the pain. Automate with purpose.**

---

## 1. Workshop purpose

This workshop helps teams identify automation opportunities using process mapping, lightweight metrics, and evidence-based prioritisation.

The aim is to move from assumption-led automation ideas to better questions, such as:

* What is actually happening?
* Where is the pain occurring?
* What evidence do we have?
* Is automation the right response?
* Is the process valuable, feasible, and ready?
* What should happen next?

The goal is not to automate everything. The goal is to identify the right opportunities and make better decisions about what to automate, improve, investigate, or defer.

---

## 2. Core idea

The workshop is built around one simple idea:

> **Map the work. Measure the pain. Automate with purpose.**

| Step                           | What it means                                                                          |
| ------------------------------ | -------------------------------------------------------------------------------------- |
| **Map the work**               | Understand how the process happens today                                               |
| **Measure the pain**           | Add evidence about effort, delay, rework, risk, or impact                              |
| **Decide the response**        | Determine whether automation, improvement, standardisation, or investigation is needed |
| **Prioritise the opportunity** | Focus on candidates that are valuable, feasible, and ready                             |

---

## 3. What makes this metrics-based?

A process map shows the flow.

Metrics show the pain.

Adding metrics helps make the invisible visible, including:

* how often the process happens
* how long it takes end to end
* how much human effort is involved
* where work waits
* where errors or rework happen
* how many handoffs are involved
* how often exceptions occur
* what systems are touched
* what risk is created
* who experiences the impact

You do not need perfect data to start. Estimates are fine as long as they are clearly marked as estimates.

---

## 4. Process mapping prompts

Use these prompts to map a process at a practical discovery level.

| Element          | Question                                              | Notes |
| ---------------- | ----------------------------------------------------- | ----- |
| **Process name** | What process are we looking at?                       |       |
| **Trigger**      | What starts the process?                              |       |
| **Outcome**      | What needs to be achieved?                            |       |
| **Steps**        | What happens from start to finish?                    |       |
| **Roles**        | Who does each step?                                   |       |
| **Systems**      | What tools, forms, spreadsheets, or systems are used? |       |
| **Decisions**    | Where are choices or approvals made?                  |       |
| **Handoffs**     | Where does work move between people or teams?         |       |
| **Exceptions**   | What happens when it deviates from the standard path? |       |

Start with the normal path first. Then add decisions, handoffs, and exceptions.

A useful map is better than a perfect map.

---

## 5. Metrics that matter

Not every process needs every metric. Choose the measures that help explain the pain and value.

### Effort

| Metric            | Question                                             | Notes |
| ----------------- | ---------------------------------------------------- | ----- |
| **Volume**        | How often does this happen?                          |       |
| **Touch time**    | How much actual human effort is involved?            |       |
| **Manual effort** | Where are people doing repetitive or low-value work? |       |

### Flow

| Metric           | Question                                                        | Notes |
| ---------------- | --------------------------------------------------------------- | ----- |
| **Cycle time**   | How long does the process take from start to finish?            |       |
| **Wait time**    | Where does work sit idle between steps?                         |       |
| **Handoffs**     | Where does work move between people or teams?                   |       |
| **Systems used** | What tools, forms, or platforms are touched across the process? |       |

### Quality and risk

| Metric              | Question                                                   | Notes |
| ------------------- | ---------------------------------------------------------- | ----- |
| **Rework / errors** | How often does work need to be corrected?                  |       |
| **Exceptions**      | How often does the process deviate from the standard path? |       |
| **Risk impact**     | What happens if something goes wrong?                      |       |
| **User impact**     | Who experiences the pain, delay, or poor outcome?          |       |

---

## 6. Time is not one thing

When looking at automation opportunities, it is useful to separate three types of time.

| Type of time   | Meaning                                 | Example                                 |
| -------------- | --------------------------------------- | --------------------------------------- |
| **Cycle time** | Total elapsed time from start to finish | A request takes five days end to end    |
| **Touch time** | Actual human effort                     | Someone spends 25 minutes processing it |
| **Wait time**  | Time sitting idle between steps         | It waits three days for approval        |

A slow process is not always an effort-heavy process.

If the issue is mostly wait time, the better response may be workflow routing, reminders, clearer approvals, better intake, fewer handoffs, or improved visibility rather than full automation.

---

## 7. Automation is one possible response

Automation is not the only possible response. The right response depends on the real problem.

| Pain point              | Possible response                                  |
| ----------------------- | -------------------------------------------------- |
| Missing information     | Better form, validation, or guidance               |
| Manual data entry       | Integration or data transfer automation            |
| Long approval delay     | Workflow routing, reminders, or escalation         |
| Repeated status chasing | Automated notifications or dashboard               |
| Frequent rework         | Standardisation, validation, or clearer rules      |
| Inconsistent decisions  | Decision tree, policy clarification, or guardrails |
| Too many exceptions     | Process redesign before automation                 |
| Poor visibility         | Tracking, reporting, or workflow transparency      |

Sometimes the best first step is to simplify, standardise, or clarify the process before automation.

---

## 8. Automation opportunity assessment

Use three lenses to assess whether an opportunity is worth taking forward.

| Lens            | Key question           | What to look for                                                 | Notes |
| --------------- | ---------------------- | ---------------------------------------------------------------- | ----- |
| **Value**       | Is this worth solving? | Time saved, risk reduced, rework reduced, better experience      |       |
| **Feasibility** | Can we automate it?    | Clear rules, stable data, system access, technical pathway       |       |
| **Readiness**   | Is the process ready?  | Process owner, agreed steps, stable process, stakeholder support |       |

A strong automation candidate is not just technically possible. It should be valuable, feasible, and ready.

---

## 9. Strong and less ready candidates

### Strong automation candidates usually have

* regular or high volume
* repetitive steps
* clear rules and decision logic
* stable inputs and outputs
* significant manual effort
* measurable errors or rework
* accessible systems or data
* clear process owner

### Less ready candidates often have

* low volume
* high judgement required
* unclear decision-making
* poor data quality
* too many exceptions
* unstable or evolving process
* unresolved policy or governance questions
* unclear ownership

Not ready for automation does not mean not worth improving.

## 11. Opportunity categories

Use these categories to decide what should happen next.

| Category                | Meaning                                      | Typical next action                                 |
| ----------------------- | -------------------------------------------- | --------------------------------------------------- |
| **Automate now**        | High value, feasible, and ready              | Move into automation discovery or design            |
| **Improve first**       | Worth solving, but process needs tidying     | Clarify, simplify, standardise, or assign ownership |
| **Investigate further** | Potential value, but more evidence is needed | Gather data or assess technical feasibility         |
| **Do not automate yet** | Low value, unstable, or unclear              | Defer or address underlying process issues          |

The output from the workshop is not a final solution. The output is the next best action.

## 14. After the workshop

After the session, the recommended next steps are:

1. Choose 2–3 candidate processes to map in more detail.
2. Validate the process with people who do the work.
3. Gather missing metrics.
4. Confirm each scenario’s category.
5. Build a small, evidence-based automation opportunity backlog.

---

## 15. Summary

The core idea is simple:

> **Map the work. Measure the pain. Automate with purpose.**

Good automation discovery starts with understanding the work, not choosing the tool.

Use the method to decide whether a process should be:

* automated now
* improved first
* investigated further
* deferred for now

The goal is to make better automation decisions based on evidence, not assumptions.
