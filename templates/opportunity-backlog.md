# Opportunity Backlog

Use this backlog to track candidate automation opportunities identified through Metrics Based Process Mapping.

The backlog is designed to help teams move from workshop discussion to follow-up action. It captures the process, pain point, evidence, likely response, opportunity category, next best action, owner and status.

> **Map the work. Measure the pain. Automate with purpose.**

---

## Purpose

This backlog helps teams:

* capture candidate automation opportunities consistently
* avoid losing useful workshop outputs
* track missing evidence and follow-up actions
* compare opportunities using value, feasibility and readiness
* identify which opportunities should move into automation discovery, process improvement or further investigation
* build a small, evidence-based automation opportunity pipeline

This is not intended to be a detailed delivery backlog. It is an early-stage discovery backlog.

---

## When to use this backlog

Use this backlog:

* after a Metrics Based Process Mapping workshop
* after a scenario clinic
* when multiple automation ideas need to be compared
* when the team needs to track missing metrics or evidence
* before deciding what moves into detailed discovery or delivery
* as a lightweight record of opportunity status and next actions

---

## How to use this backlog

1. Add each candidate process or opportunity as a row.
2. Capture the pain point in plain language.
3. Record known evidence and missing evidence.
4. Identify the likely response: automation, improvement, both, investigation or defer.
5. Categorise the opportunity.
6. Assign an owner for the next best action.
7. Review the backlog regularly and update the status.

Focus on the next best action, not the final solution.

---

# 1. Opportunity categories

Use these categories consistently.

| Category                | Meaning                                      | Typical next action                                |
| ----------------------- | -------------------------------------------- | -------------------------------------------------- |
| **Automate now**        | High value, feasible and ready               | Move into automation discovery or design           |
| **Improve first**       | Worth solving, but process needs tidying     | Clarify, simplify, standardise or assign ownership |
| **Investigate further** | Potential value, but more evidence is needed | Gather data or assess technical feasibility        |
| **Do not automate yet** | Low value, unstable or unclear               | Defer or address underlying process issues         |

Remember:

> Not ready for automation does not mean not worth improving.

---

# 2. Status values

Suggested status values:

| Status                         | Meaning                                                           |
| ------------------------------ | ----------------------------------------------------------------- |
| **New**                        | Captured but not yet reviewed                                     |
| **Needs evidence**             | Missing metrics or evidence need to be gathered                   |
| **Needs process validation**   | Current process needs to be confirmed with people who do the work |
| **Needs technical assessment** | Feasibility depends on systems, data or integration options       |
| **Improve first**              | Process improvement is required before automation                 |
| **Ready for discovery**        | Opportunity is ready for deeper automation discovery              |
| **In discovery**               | Detailed discovery or analysis is underway                        |
| **Deferred**                   | Not currently being progressed                                    |
| **Closed**                     | No further action required or opportunity has been completed      |

Teams can adapt these values to match their own delivery or governance process.

# 9. Backlog hygiene

To keep the backlog useful:

* keep entries concise
* focus on evidence, not opinions
* update missing metrics as they are gathered
* close or defer low-value items
* avoid turning every idea into a delivery item
* review categories regularly
* keep the next best action current
* assign owners for follow-up actions

A backlog is only useful if it helps the team make decisions.

---

# 10. Final reminder

The opportunity backlog is not just a list of automation ideas.

It is a record of evidence-led opportunities, assumptions, missing data and next actions.

Use it to help answer:

1. What is worth solving?
2. What is ready to automate?
3. What needs to improve first?
4. What needs more evidence?
5. What should we do next?

> **Map the work. Measure the pain. Automate with purpose.**
